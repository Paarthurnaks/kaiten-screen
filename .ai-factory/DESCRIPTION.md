# Проект: Kaiten Screen

## Обзор

Kaiten Screen — нативное десктопное приложение, автоматизирующее рутину QA-команды при заведении
задач в Kaiten. Сейчас процесс выглядит так: сделать скриншот → зайти в Kaiten → создать задачу →
прикрепить скриншот вручную. Приложение сокращает это до одного действия: пользователь снимает
скриншот выбранной области экрана, тут же заполняет короткую форму (заголовок, описание, доска/
дорожка и другие поля Kaiten), и приложение само создаёт задачу в Kaiten через API с прикреплённым
скриншотом.

Первая платформа — Windows. Архитектура закладывается так, чтобы поддержку macOS и запись видео
(вместо/вместе со скриншотом) можно было добавить позже без переписывания ядра приложения.

## Целевой сценарий использования

1. Пользователь нажимает глобальный хоткей (или иконку в трее).
2. Экран затемняется, появляется overlay для выделения произвольной области (как в стандартных
   инструментах захвата экрана).
3. После выделения области открывается компактная форма с полями задачи Kaiten: заголовок,
   описание, доска (board), дорожка/колонка (lane/column), возможно другие настраиваемые поля.
   Скриншот уже прикреплён и виден как превью.
4. По подтверждению приложение создаёт задачу в Kaiten через API (multipart-загрузка вложения)
   и показывает уведомление об успехе со ссылкой на созданную задачу.
5. Настройки приложения (API-ключ Kaiten, домен инстанса, дефолтные доска/поля, хоткеи) хранятся
   в отдельном разделе Settings.

## Ключевые функции (MVP)

- Глобальный хоткей запуска захвата области экрана (аналог Win+Shift+S, но с собственным флоу).
- Overlay выделения прямоугольной области на любом из нескольких мониторов.
- Форма создания задачи: заголовок, описание, выбор доски/дорожки и других полей Kaiten.
- Прикрепление скриншота к задаче через Kaiten API.
- Раздел настроек: API-ключ (хранится через безопасное OS-хранилище, не в открытом виде),
  адрес инстанса Kaiten, дефолтные значения полей, управление хоткеями.
- Системный трей: быстрый доступ к захвату и настройкам, приложение живёт в трее.
- Уведомления об успехе/ошибке создания задачи.

## Технологический стек

- **Платформа/рантайм:** Electron (Chromium + Node.js), TypeScript.
- **UI:** React (рендер-процесс: overlay захвата, форма задачи, окно настроек).
- **IPC:** типизированный contextBridge/IPC между main- и renderer-процессами.
- **Хранение конфигурации:** локальный JSON/файл конфигурации для не-секретных настроек;
  `safeStorage` (Electron, DPAPI на Windows) для API-ключа Kaiten.
- **HTTP-клиент:** обёртка над Kaiten REST API (создание карточки, загрузка вложения через
  multipart/form-data).
- **Сборка/дистрибуция:** electron-builder (Windows NSIS-инсталлятор на MVP, в будущем — macOS dmg).
- **Тесты:** Vitest/Jest для логики, Playwright (`_electron`) для e2e-сценариев захвата и отправки.

## Интеграция с Kaiten API

Точная схема API (эндпоинты, поля карточки, авторизация) уточняется владельцем продукта по ходу
реализации — примеры запросов/ответов будут предоставлены отдельно и заложены в реализацию клиента
API. Архитектурно клиент API оборачивается в отдельный модуль с чётким интерфейсом
(`createTask`, `attachFile`, `listBoards`, `listSpaces` и т.п.), чтоббы смену/уточнение деталей
API можно было вносить, не затрагивая UI и остальную логику приложения.

## Нефункциональные требования

- **Безопасность:** API-ключ никогда не хранится и не логируется в открытом виде; используется
  нативное безопасное хранилище ОС.
- **Устойчивость к сети:** временные сбои сети при создании задачи не должны терять сделанный
  скриншот и введённые данные формы (минимум — понятная ошибка с возможностью повторить отправку).
- **Автозапуск:** опциональный запуск приложения при старте системы и сворачивание в трей.
- **Логирование:** локальный лог-файл с ключевыми событиями (без содержимого API-ключа) для
  диагностики проблем интеграции с Kaiten.

## Будущее расширение (закладывается на уровне архитектуры, не реализуется в MVP)

- **macOS:** тот же Electron-код, отдельный overlay-слой захвата экрана и отдельная реализация
  безопасного хранилища (Keychain) под нативные различия платформы.
- **Запись видео:** дополнительный режим захвата (video capture через `desktopCapturer`/
  `MediaRecorder`) как альтернатива/дополнение к статичному скриншоту, с отдельной загрузкой
  видео-вложения в Kaiten.
- Модуль захвата (screenshot vs video) и модуль хранилища секретов (Windows vs macOS)
  проектируются через общий интерфейс с платформенными реализациями, чтобы новые платформы/режимы
  подключались модуль-за-модулем.

## Architecture

См. `.ai-factory/ARCHITECTURE.md` за детальными архитектурными гайдлайнами.
Паттерн: облегчённая Clean Architecture (порты/адаптеры вокруг захвата экрана, хранилища секретов
и клиента Kaiten API, composition root в `main/index.ts`).

## Роли в проекте

- Владелец продукта курирует проект, разбирается с деталями Kaiten API и передаёт примеры
  запросов/ответов по ходу реализации.
- Вся техническая реализация (стек, архитектура, код, дизайн UI) — на стороне AI-агента.
